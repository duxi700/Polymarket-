## 🎯 项目目标

### 核心目标
构建一个实时监控 Polymarket 市场的程序，能够：
1. ✅ 实时追踪市场价格和订单簿
2. ✅ 识别情绪驱动的价格错配（Edge）
3. ✅ 在终端输出结构化交易信号
4. ✅ 智能过滤噪声，只输出高质量信号

### 非目标
- ❌ 不自动执行交易
- ❌ 不预测比赛结果
- ❌ 不使用机器学习模型
- ❌ 不扫描整个 Polymarket 平台

## 🚧 已知限制

### 技术限制
1. **轮询模式**: 使用 HTTP 轮询而非 WebSocket
   - 延迟: 2-30 秒（取决于配置）
   - 适用场景: 非高频交易

2. **内存存储**: 使用 deque 而非数据库
   - 重启后数据丢失
   - 不支持历史回放

3. **单市场**: 当前只支持监控单个市场
   - 多市场需要多进程

### API 限制
- Polymarket API 速率限制: ~600 请求/分钟
- 当前配置: 100 请求/分钟（保守）
- 单市场 5 秒轮询: ~12 请求/分钟

### 功能限制
- 不追踪成交量（需 /trades API）
- 流动性检查简化
- 无历史数据分析

## 🎓 经验教训

### 设计决策
1. **轮询 vs WebSocket**: 选择轮询
   - 理由: 简单可靠，适合 MVP
   - 结果: 满足需求，易于维护

2. **内存 vs 数据库**: 选择内存
   - 理由: 性能高，部署简单
   - 结果: 符合实时监控场景

3. **三级信号**: 分层检测
   - 理由: 减少噪音，提高质量
   - 结果: 符合"宁可少而准"原则

### 技术选型
- **asyncio**: 异步 I/O 处理
- **Pydantic**: 数据验证和类型安全
- **Loguru**: 简单强大的日志
- **Tenacity**: 可靠的重试机制
